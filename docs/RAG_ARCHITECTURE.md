# Time-Based PD Array RAG Engine for AgentICTrader

## Executive Summary

This document proposes a **Retrieval-Augmented Generation (RAG) system** specifically designed for ICT (Inner Circle Trader) Price Action methodology. The system will augment the existing XGBoost ML pipeline with context-aware pattern matching, explainable trade reasoning, and historical precedent retrieval.

---

## 1. Core Concept: Trading Memory System

### What is it?
A vector database that stores every historical trading setup as a rich, multi-dimensional memory that can be retrieved based on similarity to current market conditions.

### Why RAG for Trading?
- **Context Preservation**: Maintains the narrative structure of price action (Accumulation → Manipulation → Expansion → Distribution)
- **Explainability**: Returns actual historical examples instead of black-box predictions
- **Adaptability**: New setups added immediately without retraining
- **Time-Aware**: Understands that a 3 AM FVG ≠ 9:30 AM FVG

## 3. Embedding Strategy

### Multi-Modal Embedding Approach

We use **multiple embedding strategies** and combine them:

#### A. Narrative Embedding (Semantic)
- Model: `sentence-transformers/all-MiniLM-L6-v2` or fine-tuned variant
- Input: The narrative description
- Captures: Semantic meaning of the setup story

#### B. Structured Feature Embedding (Numerical)
- Encode structured features into a dense vector:
  - Time window (one-hot → dense)
  - HTF structure (normalized numerical features)
  - PD array presence (binary flags)
  - Confluence factors (multi-hot encoding)
- Dimensionality reduction: PCA or autoencoder to 128-dim

#### C. Temporal Embedding
- Encode time-of-day as cyclical features:
  - `sin(2π * hour / 24)`
  - `cos(2π * hour / 24)`
- Encode day-of-week similarly
- Captures temporal patterns

#### D. Combined Embedding
```python
final_embedding = concat([
    narrative_embedding * 0.4,      # 384-dim
    structured_embedding * 0.4,     # 128-dim
    temporal_embedding * 0.2        # 16-dim
])  # Total: 528-dim
```

## 4. Retrieval Strategy

### Query Construction

When a new setup is detected:

```python
def construct_query(current_setup):
    """Build a query vector from current market state."""
    
    # 1. Generate narrative description
    narrative = generate_narrative(current_setup)
    
    # 2. Encode narrative
    narrative_emb = sentence_model.encode(narrative)
    
    # 3. Encode structured features
    structured_emb = encode_structured_features(current_setup)
    
    # 4. Encode temporal context
    temporal_emb = encode_temporal(current_setup.timestamp)
    
    # 5. Combine
    query_vector = concat([
        narrative_emb * 0.4,
        structured_emb * 0.4,
        temporal_emb * 0.2
    ])
    
    return query_vector
```

### Metadata Filtering

Before vector similarity search, apply hard filters:

```python
filters = {
    "instrument": current_setup.instrument,  # Same instrument
    "time_window": current_setup.time_window,  # Same time window
    "htf_open_bias": current_setup.htf_open_bias,  # Same HTF bias
    "outcome.result": "WIN",  # Only retrieve winning setups
    "timestamp": {"$gte": "2024-01-01"}  # Recent data (last 12 months)
}
```

### Similarity Search

```python
similar_setups = vector_store.query(
    vector=query_vector,
    filter=filters,
    top_k=10,
    similarity_metric="cosine"
)
```

### Re-Ranking

After retrieval, re-rank by:
1. **Outcome quality**: Higher R-multiple setups ranked higher
2. **Recency**: More recent setups weighted higher
3. **Confluence similarity**: Exact confluence factor matches boosted

```python
def rerank_score(retrieved_setup, current_setup):
    base_score = retrieved_setup.similarity_score
    
    # Boost for high R-multiple
    r_boost = min(retrieved_setup.outcome.r_multiple / 5.0, 1.0) * 0.2
    
    # Boost for recency (exponential decay)
    days_old = (now - retrieved_setup.timestamp).days
    recency_boost = exp(-days_old / 90) * 0.15  # 90-day half-life
    
    # Boost for exact confluence match
    confluence_overlap = len(set(retrieved_setup.confluence_factors) & 
                             set(current_setup.confluence_factors))
    confluence_boost = (confluence_overlap / 5.0) * 0.15
    
    return base_score + r_boost + recency_boost + confluence_boost
```

def generate_trade_reasoning(current_setup, similar_setups):
    """Generate explainable reasoning using retrieved examples."""
    
    # Build context from retrieved setups
    context = ""
    for i, setup in enumerate(similar_setups[:3], 1):
        context += f"""
        Example {i} ({setup.timestamp.date()}):
        - Setup: {setup.narrative}
        - Outcome: {setup.outcome.r_multiple}R ({setup.outcome.result})
        - Duration: {setup.outcome.duration_minutes} minutes
        """
    
    # LLM prompt
    prompt = f"""
    You are an expert ICT trader. Generate trade reasoning for the current setup.
    
    Current Setup:
    - Instrument: {current_setup.instrument}
    - Time: {current_setup.timestamp} ({current_setup.time_window})
    - HTF Bias: {current_setup.htf_open_bias}
    - PD Arrays: {current_setup.pd_arrays}
    - Entry: {current_setup.entry_price}
    - SL: {current_setup.stop_loss}
    - TP: {current_setup.take_profit}
    
    Similar Historical Setups:
    {context}
    
    Generate reasoning using the 3-question framework:
    1. Where has price come from?
    2. Where is it now?
    3. Where is it likely to go?
    
    Reference the historical examples to support your reasoning.
    """
    
    reasoning = llm.generate(prompt)
    return reasoning

## 7. Vector Store Technology Stack

### Recommended: Qdrant

**Why Qdrant?**
- Native support for metadata filtering + vector search
- High performance (Rust-based)
- Easy Docker deployment
- Python SDK with async support
- Supports payload-based filtering before vector search

### Alternative: Pinecone

**Pros:**
- Managed service (no infrastructure)
- Excellent performance
- Good documentation

**Cons:**
- Cost (paid service)
- Vendor lock-in

### Schema in Qdrant

## 13. Risks & Mitigation

### Risk 1: Overfitting to Historical Patterns
**Mitigation**: 
- Use recency weighting
- Require minimum sample size (e.g., 5 similar setups)
- Fall back to ML-only if retrieval confidence is low

### Risk 2: Retrieval Latency
**Mitigation**:
- Use approximate nearest neighbor (ANN) algorithms
- Cache frequent queries
- Pre-compute embeddings for common setups

### Risk 3: Data Quality
**Mitigation**:
- Manual review of first 100 setups
- Automated validation (e.g., OHLC constraints)
- Continuous monitoring of retrieval quality

### Risk 4: LLM Hallucination
**Mitigation**:
- Ground all reasoning in retrieved examples
- Validate cited examples exist
- Fall back to template-based reasoning if LLM fails
